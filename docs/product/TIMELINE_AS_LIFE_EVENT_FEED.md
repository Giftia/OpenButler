# Timeline as Life Event Feed

OB-GOAL-013 把 `/timeline` 当作 OpenButler 的信任资产来做。用户不是来看日志的，他们要知道发生了什么、这条记录从哪来、能不能相信。

## 信息层级

单条事件按这个顺序呈现：

1. 时间，放轻一点，只做索引。
2. 事件标题和摘要，这是视觉重点。
3. 状态：待处理、可复核、已记录。
4. 来源、类型和证据提示。
5. 右侧缩略图或证据占位。

时间、来源、可信度不抢标题。用户先扫事件，再决定要不要看依据。

## 筛选

时间线保留三个筛选：

- 时间：今天、昨天、近 7 天、全部。
- 分类：工作、物品、提醒、家庭、健康习惯、自动化。
- 重要性：全部、值得处理、仅记录、有依据。

筛选无结果时，不显示空表格。文案是：

> 这个筛选下暂时没有事件。可以放宽时间、分类或重要性条件，再看看。

## 缩略图和证据占位

事件卡右侧必须有稳定的视觉锚点。

- 有安全图片 URL 时显示缩略图。
- 只有本地截图路径时，不加载、不展示路径，只显示“有本地截图依据”。
- 没有图片时，用来源占位。
- 样例事件可以显示样例占位，但不能伪装成真实用户数据。

普通 UI 不显示 `C:\Users\...`、`screenshots`、`raw_ref`、`source_event_id`、`evidence_refs`。

## 依据抽屉

用户点击“查看依据”后，只显示这些内容：

- 来源
- 可信度
- 边界说明
- 隐私说明

文案要清楚，但不要像调试输出：

> 这只是线索，不代表外部服务的真实状态。

> 不会显示本地截图路径。

## 本轮边界

本轮只改前端体验、任务队列和产品文档。

- 不新增后端 API。
- 不新增 `/api/butler/insights/{id}/evidence`。
- 不读取真实 MineContext 数据。
- 不复制截图。
- 不调用外部模型。
