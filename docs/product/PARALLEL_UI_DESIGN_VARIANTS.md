# Parallel UI Design Variants

OpenButler 当前的主要问题不是缺功能，而是普通用户很难马上看懂这些功能和自己有什么关系。继续在旧首页上细修，容易把技术原型越修越复杂。因此本轮新增设计实验室，用同一套现有接口做三种独立视觉方向，再用中立测试判断主线应该收敛到哪一种。

## 入口

- `/design-lab`：设计实验室总览。
- `/design/mijia`：米家式状态中控。
- `/design/ios`：iOS / Apple Home 式私人管家。
- `/design/deck`：大屏 PPT 式商业演示版。

这些页面只用于体验探索，不替换当前 `/butler` 主路径。设计实验页可以绕过首次激活门禁，方便测试；普通主路径仍要求完成引导后再进入。

## 三种方向

### 米家式状态中控

目标是让用户先看到状态和数字，而不是模块。页面使用大卡片、场景分组和少量标签，适合回答：今天发生了什么、哪件事值得先看、还有哪些场景信号。

### iOS / Apple Home 式私人管家

目标是降低长期使用压力。页面保留更多留白和克制卡片，强调一个主行动和几条清晰记录，适合普通用户日常打开。

### 大屏 PPT 式演示版

目标是商业评估和讲故事。页面用大数字和强对比结构，把 OpenButler 的价值讲成一页产品说明，而不是日常操作台。

## 测试问题

每次修复后，新的中立测试员只从使用者视角回答：

1. 这是什么软件？
2. 是否知道它能干什么？
3. 是否会配置？
4. 会怎么用？
5. 现在会用了吗？
6. 觉得有用吗？

测试员不应看代码，也不应被告知 OpenButler 的产品定义。结论用于判断哪种设计最能让新用户理解价值。

## 文案边界

普通用户界面不出现 `MineContext`、`PCActivity`、`UnifiedTimelineEvent`、`mock`、`seed`、`Provider`、`Webhook` 等内部词。本机真实能力统一说成“本机记录组件”和“智能整理能力”；底层项目名只放在高级说明里。

## 当前限制

- 三套版本复用现有 API 和样例数据，不新增后端能力。
- 当前没有替换主导航首屏，必须经过中立测试后再决定收敛方向。
- 浏览器自动化可能受本机安全策略限制；如果无法跑自动 smoke，需要人工打开以上路由检查。
